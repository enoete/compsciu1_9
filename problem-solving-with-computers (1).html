üí° What is Problem Solving?

Problem solving can be described as a systematic approach or process to resolve a problem using a computerised solution. It is a cognitive process that can be best achieved by following a framework of activities.

üéØ Key Insight: Problem-solving is an art but it also requires a logical mind!

Why Use Computers for Problem Solving?

A computer system is built upon a hardware system and only becomes operational with software. Software is a collection of coded statements defined by a particular programming language, designed for a specific purpose.

üéÆ Click to Explore: Types of Software
Application Software ‚ñº

Completes certain tasks and interacts with users of a computer system. Examples include word processors, web browsers, and games.

System Software ‚ñº

Interacts with the hardware of the computer system. Examples include operating systems and device drivers.

üìå Important: In the development of software, a rigorous approach must be taken to ensure that it is error-free and behaves in a predictable manner. This is why we need a structured approach to problem solving!

üîÑ The Problem-Solving Process

Click on each step to learn more about it:

1. Problem Definition
The first step is to be clear on what the problem is to be solved. The issue must be explicitly stated, the boundaries conceptualised, and unambiguously stated. If there is no clear definition, no correct solution can be created!
‚Üí
2. Problem Analysis
Here we analyze what data we have been given and how we can use that data to achieve the goal. We also look at the processing requirements to transform the inputs into the required outputs.
‚Üí
3. Identify Solutions
After analysis, find solutions to the problem. This may involve looking at solutions that were already built for similar problems or developing your own solution.
‚Üí
4. Select Optimal Solution
Rate each possible solution in a comparison chart to check for suitability, scope, cost, and stakeholder satisfaction. The selection may depend on factors such as affordability.
‚Üí
5. Implementation
This involves representing the solution as an algorithm and developing the algorithm into a functioning piece of software.
‚Üí
6. Review
The completed solution is tested for correctness to see if it meets the requirements outlined by the stakeholders.

üß© Modular Programming (Divide and Conquer)

Also called Top-Down Design, this approach involves continuously breaking or dividing a problem into successively smaller problem sets or sub-problems.

Benefits of Modular Programming:

  • Problem solution becomes more controllable
  • Easier to maintain and test
  • Simpler logical steps for each sub-problem
  • Less complex logic overall

Two Key Features:

Cohesion

The arrangement of functions or modules in a system in relation to performing a particular task well.

Coupling

A measure of how interconnected or functionally dependent the modules are within a system.

‚ö° Algorithms

An algorithm is a finite set of precise instructions for solving a problem in a fixed amount of time. Programs are clear sets of defined, unambiguous instructions given to the computer to follow to perform a specific task.

Properties of an Algorithm

üéØ

Precise

Clear and exact instructions

üîç

Unambiguous

Only one interpretation possible

‚è±Ô∏è

Finite

Terminates after a finite number of steps

üìã

Logical Sequence

Steps follow a logical order

Three Stages of Creating an Algorithm

  1. Determining the input data - What information is needed?
  2. Determining the processing requirements - What operations need to be performed?
  3. Outputting the processed input - What results should be displayed?

üìù Algorithm Structure

Every algorithm should have these sections in order:

  1. Header - The algorithm's name or title
  2. Variable Declaration - Brief description and variables that will be used
  3. Program Body - Sequence of steps to accomplish the output
  4. Terminator - An end statement marking the end of the algorithm

Methods for Writing Algorithms

1. Narrative

A series of well-defined sentences describing the algorithm in precise language.

2. Pseudocode

Statements resembling natural language using similar principles as programming - a mid-point between narrative and actual code.

3. Flowchart

A graphical representation using special symbols to represent each step and directed paths.

Input, Processing, and Output

üîÄ Control Structures

A control structure is a basic component used to express programming logic. Programs are developed from:

  • Input Statements
  • Output Statements
  • Control Structures (Sequence, Selection, Iteration)
  • Assignment Statements

Selection Control Structure

The selection control structure permits the execution of particular actions based on given criteria.

IF-THEN Construct ‚ñº

A statement that presents a condition that will be evaluated as true or false, and based on the result, the appropriate action is taken.

IF <Condition> THEN <instructions that will be carried out if the condition is true> END IF // Example: IF Number > 5 THEN Sum ‚Üê number * number END IF
IF-THEN-ELSE Construct ‚ñº

The ELSE segment is executed should the condition be tested and found false.

IF <Condition> THEN <statements that will be carried out if the condition is true> ELSE <statements that will be carried out if the condition is false> END IF // Example: IF Number < 0 THEN neg_num ‚Üê neg_num + 1 ELSE pos_num ‚Üê pos_num + 1 ENDIF
Nested IF Statements ‚ñº

A nested IF statement is a statement where an IF statement is within another IF statement.

IF <condition1> THEN statement 1 ELSE IF <condition2> THEN statement 2 ELSE IF <condition3> THEN statement 3 ELSE statement 4 ENDIF ENDIF ENDIF
CASE Statements ‚ñº

A selection structure used to give a finite set of options, accepting various inputs from a user, and providing a detailed set of operations for each option presented.

CASE <Variable> C1: <Statements to be executed> C2: <Statements to be executed> C3: <Statements to be executed> C4: <Statements to be executed> Default: <Statements if an unacceptable input is entered> ENDCASE

üîÅ Iterative Structures (Loops)

Loops are used for generating cumulative totals or for counting how many times the looping structure has repeated itself. A loop needs to know:

  • What to repeat
  • How often to repeat the steps

Types of Loops:

  • WHILE loop - Condition-controlled (unbounded)
  • FOR loop - Counter-controlled (bounded)
  • REPEAT UNTIL - Condition-controlled (executes at least once)

üîπ Bounded vs Unbounded Loops

Bounded (Finite): A loop that will repeat a known number of times, such as the FOR loop.

Unbounded (Infinite): A loop where the number of iterations is not known, such as the WHILE loop.

WHILE Loop

Executes a series of statements as long as a given condition is True.

WHILE <Condition> DO <Action to be taken if condition is true> ENDWHILE // Example: WHILE Value <> -1 DO <Actions to be repeated> ENDWHILE

FOR Loop

Repeats a group of statements a specified number of times. Used when the exact number of repetitions is known.

FOR <Variable> = <Beginning value> TO <Ending value> DO <Actions to be repeated> ENDFOR // Example: FOR tally = 1 TO 10 DO PRINT "Enter a value greater than 4" READ value ENDFOR

üìå FOR Loop Terminology

  • Counter: Numeric variable used as a loop counter (can't be an array element)
  • Start: Initial value of counter
  • End: Final value of counter
  • Statements: One or more statements between FOR and ENDFOR that are executed the specified number of times

üìã System Requirements

When developing software solutions, it's essential to understand what the system must do and how well it must perform. Requirements are divided into two main categories: Functional and Non-Functional requirements.

üí° Why Requirements Matter: Clear requirements help developers understand exactly what needs to be built and ensure that the final product meets user expectations. Without proper requirements, projects can fail or deliver solutions that don't solve the actual problem!

Functional Requirements

üìå Definition

Functional requirements describe WHAT a system should do. They specify the specific behaviors, functions, and features that the system must have. These are the core capabilities that directly address user needs.

Key Characteristics:

  • Describe specific system behaviors and functions
  • Define what inputs the system should accept
  • Specify what outputs the system should produce
  • Detail how the system should respond to particular situations
  • Can be tested with specific test cases (pass/fail)

üéØ Examples of Functional Requirements

üè¶

Banking System

  • The system must allow users to check account balance
  • The system must allow users to transfer funds between accounts
  • The system must generate monthly statements
  • Users must be able to change their PIN
üõí

Online Store

  • Customers must be able to add items to a shopping cart
  • The system must calculate total cost including tax
  • Users must be able to search for products by name
  • The system must send order confirmation emails
üìö

Library System

  • The system must allow users to search for books
  • Users must be able to reserve books online
  • The system must track borrowed books and due dates
  • Librarians must be able to add new books to the catalog
üéì

School Management

  • Teachers must be able to enter student grades
  • The system must calculate student averages
  • Parents must be able to view their child's attendance
  • The system must generate report cards

Non-Functional Requirements

üìå Definition

Non-functional requirements describe HOW WELL a system should perform. They specify the quality attributes, constraints, and standards that the system must meet. These requirements focus on the overall behavior and characteristics of the system rather than specific functions.

Key Characteristics:

  • Define quality attributes of the system
  • Specify performance standards and constraints
  • Address security, reliability, and usability concerns
  • Often measured in degrees (e.g., response time < 2 seconds)
  • Apply across the entire system, not just specific features

üéØ Categories of Non-Functional Requirements

‚ö°

Performance

üîí

Security

Protection of data and prevention of unauthorized access

Example: "All passwords must be encrypted"

üîÑ

Reliability

System's ability to function without failure

Example: "The system must have 99.9% uptime"

üìà

Scalability

Ability to handle growth in users or data

Example: "System must support 10,000 concurrent users"

üë§

Usability

How easy the system is to use and learn

Example: "New users must complete tasks without training"

üîß

Maintainability

How easy it is to modify and update the system

Example: "Code must follow documentation standards"

üíª

Portability

Ability to work on different platforms/environments

Example: "Must work on Windows, Mac, and Linux"

‚ôø

Accessibility

Usable by people with disabilities

Example: "Must support screen readers"

‚ö†Ô∏è Common Mistake
